## 二、环境要求

### 本地开发机（打包用）
| 工具 | 版本要求 | 用途 |
|------|---------|------|
| Node.js | **20.18+** | 运行 npm / vite / tsc |
| npm | 10.x | 包管理 |
| Git | 任意版本 | 代码管理（可选） |
| Python | 3.x（仅 Windows 全新机器需要） | better-sqlite3 编译兜底 |
| VS Build Tools | 2019+（仅 Windows 全新机器需要） | better-sqlite3 编译兜底 |

> better-sqlite3 12.x 在 npm 上自带 **Windows x64 / Linux x64 / macOS x64+arm64** 的预编译二进制（prebuild），绝大多数情况下 `npm install` 直接下载 `.node` 文件即可，**无需本机编译**。仅当预编译下载失败时才会 fallback 到本地用 node-gyp 编译，这时才需要 Python + 编译工具链。

### 服务器（部署目标）
| 工具 | 版本要求 | 用途 |
|------|---------|------|
| Linux x64（CentOS / Rocky / Ubuntu / Alibaba Cloud Linux） | 任意主流版本 | 操作系统 |
| Node.js | **20.18+** | 运行后端服务（better-sqlite3 12.x 要求 Node ≥ 20） |
| PM2 | 5.x+ | Node 进程守护 |
| gcc / g++ / make | 任意版本 | better-sqlite3 编译兜底（仅预编译下载失败时用） |
| nginx（可选） | 1.20+ | 反向代理 / HTTPS |

> ⚠️ **本次升级与 v1.1 的关键差异**：
> 后端从纯 JS 的 `sql.js` 切换到了 **native 模块 `better-sqlite3`**，包含 C++ 编译产物。这意味着：
> 1. 服务器必须**重装** node_modules（不能复用 Windows 的）
> 2. 服务器需要 Linux x64 架构（普通云主机都满足）
> 3. 万一拉不到预编译二进制，需要本地 `gcc / g++ / make` 兜底编译

### 服务器初始化（首次部署时执行一次）

```bash
# === 1) 系统基础工具 ===
dnf makecache
dnf install -y wget curl

# === 2) 编译工具链（better-sqlite3 兜底用，强烈建议装上）===
# CentOS / Rocky / Alibaba Cloud Linux
sudo dnf install -y gcc gcc-c++ make
# Ubuntu / Debian
# sudo apt-get update && sudo apt-get install -y build-essential

# === 3) Node.js 20 ===
wget https://npmmirror.com/mirrors/node/v20.18.0/node-v20.18.0-linux-x64.tar.xz
tar -xf node-v20.18.0-linux-x64.tar.xz -C /usr/local/
mv /usr/local/node-v20.18.0-linux-x64 /usr/local/node
echo "export PATH=/usr/local/node/bin:\$PATH" >> /etc/profile
source /etc/profile

node -v   # 应显示 v20.18.0
npm -v    # 应显示 10.x

# === 4) npm 镜像加速（强烈推荐，否则 better-sqlite3 预编译下载会很慢）===
npm config set registry https://registry.npmmirror.com

# === 5) PM2 ===
npm install -g pm2 --registry=https://registry.npmmirror.com
pm2 -v

# === 6) PM2 开机自启 ===
pm2 startup     # 执行后会输出一条命令，复制粘贴执行即可
pm2 save
```

## 三、首次部署完整流程

### Step 1：本地打包

#### Windows
```bat
:: 进入项目根目录
cd C:\path\to\xihualvye

:: 双击或命令行执行
build.bat
```

#### Linux / Mac
```bash
cd /path/to/xihualvye
chmod +x build.sh
./build.sh
```

执行完成后，会生成两个关键产物：
- `server/public/`     — 前端静态文件
- `server/dist/`       — 后端编译产物

> ⚠️ 本地的 `server/node_modules/better-sqlite3/build/Release/better_sqlite3.node` 是 **Windows 二进制**，绝对不能上传到 Linux 服务器使用。我们的部署流程本来就只上传 `dist/` 和 `public/`，不上传 `node_modules`，所以这点天然规避。

### Step 2：上传文件到服务器

使用 **XFTP / WinSCP / FileZilla** 上传到服务器目录（以 `/xmc/xihualvye/` 为例）：

| 必传 | 路径 | 说明 |
|:---:|------|------|
| ✅ | `server/dist/` | 后端 JS（编译产物） |
| ✅ | `server/public/` | 前端静态文件 |
| ✅ | `server/package.json` | 依赖清单（**v1.2 已切换为 better-sqlite3，必须更新**） |
| ✅ | `server/package-lock.json` | 依赖锁定文件（保证版本一致） |

| 严禁上传 | 路径 | 说明 |
|:---:|------|------|
| ❌ | `server/data/` | 服务器上的生产数据库，**覆盖会丢全部订单数据** |
| ❌ | `server/node_modules/` | Windows 的 native 二进制 ≠ Linux，必须服务器重装 |
| ❌ | `server/src/` | 源代码服务器不需要 |

> **首次部署时**：服务器上还没有 `data/` 目录，无需担心覆盖。后续更新时严禁上传本地的 `data/`。

### Step 3：服务器安装依赖

```bash
ssh user@your-server-ip
cd /xmc/xihualvye

# 安装生产依赖（首次或 package.json 有变化时）
# better-sqlite3 12.x 会自动从 npm 镜像拉取 Linux x64 预编译二进制
npm install --production
```

**预期看到的关键日志**：
```
> better-sqlite3@12.9.x install
> prebuild-install || node-gyp rebuild --release
prebuild-install info looking for cached prebuild ...
prebuild-install info found prebuild in cache, unpacking
```

**装完验证**：
```bash
ls -la node_modules/better-sqlite3/build/Release/better_sqlite3.node
# 应该看到这个 .node 二进制文件存在（约 2-4MB）
```

如果看到 `prebuild-install warn install No prebuilt binaries found`，会自动 fallback 到 `node-gyp rebuild`，需要 1-2 分钟。这就是为什么前面要装 `gcc gcc-c++ make`。

### Step 4：配置环境变量（强烈推荐）

为了 token 在服务重启后不失效、以及修改默认账号密码，建议通过环境变量启动：

```bash
# 生成一个 64 位随机密钥（执行一次，记下来）
openssl rand -hex 32
# 输出类似：4f8a2d9b3c1e7f6a8d2b4c5e7f9a1b3d5c7e9f1a3b5c7d9e1f3a5b7c9d1e3f5a
```

支持的环境变量：

| 变量名 | 默认值 | 说明 |
|---|---|---|
| `PORT` | 3456 | 后端监听端口 |
| `ADMIN_USERNAME` | xhly | 登录账号 |
| `ADMIN_PASSWORD` | 123123 | 登录密码（**生产环境强烈建议改**） |
| `TOKEN_SECRET` | （随机生成） | Token 签名密钥。不设置时每次重启服务会随机生成，导致已登录用户被强制重登 |
| `DB_DIR` | `<dist>/../data` | 数据库文件目录 |

### Step 5：首次启动 & 观察迁移日志

**方式 A：临时单条命令启动（适合测试）**
```bash
TOKEN_SECRET=4f8a2d9b... \
ADMIN_PASSWORD=改成你自己的强密码 \
pm2 start dist/index.js --name xihualvye --update-env
```

**方式 B：用 ecosystem 配置文件启动（推荐生产）**

创建 `/xmc/xihualvye/ecosystem.config.js`：

```js
module.exports = {
  apps: [{
    name: 'xihualvye',
    script: './dist/index.js',
    cwd: '/xmc/xihualvye',
    instances: 1,
    exec_mode: 'fork',
    autorestart: true,
    max_memory_restart: '500M',
    env: {
      NODE_ENV: 'production',
      PORT: 3456,
      ADMIN_USERNAME: 'xhly',
      ADMIN_PASSWORD: '改成你自己的强密码',
      TOKEN_SECRET: '4f8a2d9b3c1e7f6a8d2b4c5e7f9a1b3d5c7e9f1a3b5c7d9e1f3a5b7c9d1e3f5a'
    }
  }]
}
```

启动：
```bash
cd /xmc/xihualvye
pm2 start ecosystem.config.js
pm2 save
```

**首次启动预期日志**（含自动迁移）：
```
📁 数据库路径: /xmc/xihualvye/data/xihualvye.db
🔧 检测到 quantity 字段为旧的 TEXT 类型，开始迁移为 INTEGER ...
✅ quantity 字段迁移完成（TEXT → INTEGER）
✅ 数据库初始化完成
📦 生产模式：已加载前端静态文件
✅ 玺华铝业后端服务已启动: http://localhost:3456
```

> 老库自动迁移说明：
> - 若库里 `quantity` 是 TEXT 类型，会一次性迁移为 INTEGER（**不可逆，迁移前务必备份**）
> - 若缺少 `status` 列，会自动 ALTER TABLE 添加，默认 `pending`
> - 若有 `shipped` 状态的历史订单，会迁移为 `completed`

### Step 6：访问验证

浏览器打开 `http://your-server-ip:3456` → 看到登录页 → 输入账号 `xhly / <你设置的密码>` → 进入系统首页 → 见到统计卡片和图表说明部署成功。

F12 打开 Network 面板检查：
- 业务接口的 Request Headers 里应有 `Authorization: Bearer xxxxx`
- 不应再频繁出现 `/auth/verify` 请求（v1.2 已移除每次请求前的预校验）
- `GET /api/orders/stats/dashboard?days=7` 应一次返回所有首页统计数据

## 四、从 v1.1（sql.js）升级到 v1.2（better-sqlite3）的特别流程

> 如果你的服务器上已经在跑 v1.1 老版本，按这份流程升级。**最关键的是先备份再说话**。

### Step 1：服务器先备份（最关键的一步）

```bash
ssh user@your-server-ip
cd /xmc/xihualvye

# 1) 备份数据库（quantity 字段迁移不可逆）
mkdir -p /xmc/backup
cp data/xihualvye.db /xmc/backup/xihualvye_before_v1.2_$(date +%Y%m%d_%H%M%S).db

# 2) 备份当前代码产物（万一要回滚）
TS=$(date +%Y%m%d_%H%M%S)
cp -r dist /xmc/backup/dist_$TS
cp -r public /xmc/backup/public_$TS
cp package.json /xmc/backup/package.json_$TS
cp package-lock.json /xmc/backup/package-lock.json_$TS
cp -r node_modules /xmc/backup/node_modules_$TS    # 含旧的 sql.js

ls -la /xmc/backup/  # 确认备份成功
```

### Step 2：上传新版文件（同三.Step 2）

⚠️ 必须同时上传新的 `package.json` 和 `package-lock.json`，否则装出来还是 sql.js。

### Step 3：清理旧依赖 & 重装

```bash
cd /xmc/xihualvye

# 先停服务
pm2 stop xihualvye

# 删除老 node_modules（含旧 sql.js，必须清掉）
rm -rf node_modules

# 重装（会装上 better-sqlite3 12.x）
npm install --production
```

### Step 4：配置 TOKEN_SECRET 并启动

参照本文档「三.Step 4 / Step 5」配置环境变量并启动。

### Step 5：浏览器强制刷新 & 重新登录

老用户的 localStorage 里没有新版 token，第一次访问会被自动跳到登录页，重新登录一次即可。

```
Ctrl + F5（必须）→ 自动跳转登录页 → 用新密码登录 → 验证 Dashboard
```

### Step 6：观察 5-10 分钟

```bash
pm2 monit                          # 看 CPU / 内存
pm2 logs xihualvye --lines 100     # 看是否有报错
```

正常指标：
- 内存：30-80MB（v1.1 时常常 100-300MB）
- CPU：空闲时 0%
- 没有 `Error:` 之类堆栈

## 六、数据库说明

### 数据存储位置
- 路径：`server/data/xihualvye.db`
- 引擎：**better-sqlite3 12.x**（同步、WAL 模式，性能比 sql.js 高 10-50 倍）
- 大小：通常几 MB ~ 几十 MB
- 关联文件：除了 `xihualvye.db`，WAL 模式还会产生 `xihualvye.db-wal` 和 `xihualvye.db-shm` 两个临时文件，**备份和恢复时三个文件最好一起处理**（或先 stop 服务让 WAL 落盘，再仅备份 .db）

### 自动迁移机制
本系统**已内置自动迁移**：每次启动时检查表结构，发现以下情况会自动处理：
- `quantity` 字段为 TEXT → 自动迁移为 INTEGER（一次性，不可逆）
- 缺少 `status` 列 → 自动 ALTER TABLE 添加，默认 `pending`
- 历史 `shipped` 状态订单 → 自动迁移为 `completed`

> **升级到 v1.2 后首次启动**，控制台会打印类似：
> ```
> 🔧 检测到 quantity 字段为旧的 TEXT 类型，开始迁移为 INTEGER ...
> ✅ quantity 字段迁移完成（TEXT → INTEGER）
> ```

### 备份建议（极重要）

**强烈建议设置定时备份**，否则磁盘损坏 = 数据全丢。

#### 方案 A：crontab 自动备份（推荐）
```bash
crontab -e

# 每天凌晨 2 点备份一次，保留近 30 天
0 2 * * * cp /xmc/xihualvye/data/xihualvye.db /xmc/backup/xihualvye_$(date +\%Y\%m\%d).db && find /xmc/backup -name "xihualvye_*.db" -mtime +30 -delete
```

> WAL 模式下，运行中的 `.db` 文件可能不包含最新写入（最新数据在 `.db-wal` 里），better-sqlite3 启动时会用 `synchronous = FULL`，每次提交都会刷盘并 checkpoint，但保险起见，重要时刻可以先 `pm2 stop xihualvye` 再备份。

#### 方案 B：手动备份
```bash
cp /xmc/xihualvye/data/xihualvye.db /xmc/backup/xihualvye_$(date +%Y%m%d).db
```

### 数据恢复
```bash
pm2 stop xihualvye
# 删掉可能存在的 WAL 临时文件
rm -f /xmc/xihualvye/data/xihualvye.db-wal /xmc/xihualvye/data/xihualvye.db-shm
cp /xmc/backup/xihualvye_20260401.db /xmc/xihualvye/data/xihualvye.db
pm2 start xihualvye
```

## 九、可选：用 nginx 反向代理 + HTTPS

若需要域名访问 + HTTPS（强烈推荐生产环境使用）：

### 1. 安装 nginx
```bash
dnf install -y nginx
```

### 2. 配置反向代理

```nginx
cat > /etc/nginx/conf.d/xihualvye.conf << 'EOF'
server {
    listen 80;
    server_name your-domain.com;

    location / {
        proxy_pass http://127.0.0.1:3456/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
EOF
```

### 3. 测试并重启
```bash
nginx -t
systemctl restart nginx
systemctl enable nginx
```

### 4. 申请免费 HTTPS 证书
```bash
sudo dnf install -y certbot python3-certbot-nginx
sudo certbot --nginx -d your-domain.com
```

## 十一、常见问题排查

### Q1. 启动报错 `Error: Cannot find module 'better-sqlite3'`
原因：依赖未安装。
```bash
cd /xmc/xihualvye
npm install --production
pm2 restart xihualvye
```

### Q2. 启动报错 `Error: ... better_sqlite3.node ... invalid ELF` 或 `is not a valid Win32 application`
原因：把 Windows 的 node_modules 上传到 Linux 了，二进制不兼容。
```bash
cd /xmc/xihualvye
rm -rf node_modules
npm install --production
pm2 restart xihualvye
```

### Q3. `npm install` 卡在 `prebuild-install` 或 `node-gyp rebuild`
原因：拉不到预编译二进制，且本机没有编译工具链。
```bash
# CentOS / Rocky
sudo dnf install -y gcc gcc-c++ make
# Ubuntu / Debian
sudo apt-get install -y build-essential

# 切换 npm 镜像后重试
npm config set registry https://registry.npmmirror.com
rm -rf node_modules
npm install --production
```

### Q4. 浏览器访问后一直跳回登录页
原因：localStorage 里的旧 token 与新 `TOKEN_SECRET` 不匹配（升级后正常现象）。
解决：F12 → Application → Local Storage → 清空 → 重新登录。

### Q5. 服务重启后所有人都被踢下线
原因：没有设置 `TOKEN_SECRET`，每次启动会随机生成新 secret。
解决：参照「三.Step 4」配置固定的 `TOKEN_SECRET`。

### Q6. 浏览器访问空白
1. F12 看控制台报错
2. 确认上传了 `server/public/` 目录
3. 检查 nginx 配置是否正确（如使用代理）
4. 强制刷新 Ctrl+F5

### Q7. 数据库文件锁住 `database is locked`
better-sqlite3 是单写多读，且使用 WAL，几乎不会锁。如果出现：
```bash
pm2 stop xihualvye
sleep 5
ls -la /xmc/xihualvye/data/   # 查看是否有遗留的 .db-wal / .db-shm
pm2 start xihualvye
```

### Q8. 想看实时性能
```bash
pm2 monit
```

### Q9. 服务器重启后服务没自动起来
```bash
pm2 startup    # 查看提示并执行其输出的命令
pm2 save       # 保存当前服务列表
```

## 十二、回滚方案

### 紧急回滚到上一版本（v1.2 → v1.1）

> 前提：升级前已按「四.Step 1」完整备份。

```bash
cd /xmc/xihualvye
pm2 stop xihualvye

# 1. 恢复数据库（最关键）
rm -f data/xihualvye.db-wal data/xihualvye.db-shm
cp /xmc/backup/xihualvye_before_v1.2_<时间戳>.db data/xihualvye.db

# 2. 恢复代码产物
TS=<你的备份时间戳>
rm -rf dist public node_modules package.json package-lock.json
cp -r /xmc/backup/dist_$TS dist
cp -r /xmc/backup/public_$TS public
cp /xmc/backup/package.json_$TS package.json
cp /xmc/backup/package-lock.json_$TS package-lock.json
cp -r /xmc/backup/node_modules_$TS node_modules

# 3. 启动旧版（v1.1 不需要 TOKEN_SECRET）
pm2 start xihualvye
pm2 logs xihualvye --lines 30
```
