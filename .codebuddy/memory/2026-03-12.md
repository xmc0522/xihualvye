## 修复排除名称后配件值（上包转角/下包转角）不联动更新的问题

### 问题根本原因（二次修复）：
防抖 watch 有 300ms 的 setTimeout 延迟，但配件计算 watch 监听 filteredTableData 时会**立即**触发。此时防抖的 setTimeout 还未执行，tableData 中的 shuliang 仍是旧值，所以配件值读到的还是原来的数字。

### 最终解决方案：
1. **直接检查排除列表**：配件计算不再依赖防抖后 tableData 中被赋0的值，而是直接检查 `excludeNamesRef.value.includes('上包边')` 来判断是否被排除
2. **watch 监听源改为 `excludeNamesRef.value`**：替代原来的 `filteredTableData.value`，确保排除列表变化时立即触发，无时序问题
3. 被排除时配件值赋 `'0'`

### 关键教训：
防抖 setTimeout 修改数据 + watch 依赖该数据 = 时序 bug。应直接在计算逻辑中检查排除状态。

### 文件变更：
- `xihualvye/src/ts/自由选择-单面门-通配款/自由选择-单面门-通配款.ts` - 配件计算直接检查 excludeNamesRef 而非依赖防抖后的 tableData

## 修复配件表（上包转角/下包转角）页面不更新的问题（三次修复）

### 问题根本原因：
`allAccessories` 是一个**普通数组**（非响应式），当 watch 中修改 `shangBaoZhuanJiao.value = '0'` 时，虽然数据确实被修改了，但 Vue **无法检测到变化**。`accessoryRows` 是 `computed`，它依赖 `allAccessories`，但由于 `allAccessories` 不是响应式的，computed 不会重新计算，页面不会更新。

### 解决方案：
将 `allAccessories` 从普通数组改为 `reactive` 响应式数组：`reactive<{ name: string; value: string }[]>([...])`

### 关键教训：
Vue 的 `computed` 只有在依赖的**响应式数据**变化时才会重新计算。普通数组/对象即使被修改，computed 也不会感知到。凡是在 watch/computed 中会被修改的数据，都必须是 reactive/ref。

### 文件变更：
- `xihualvye/src/ts/自由选择-单面门-通配款/自由选择-单面门-通配款-配件数据.ts` - allAccessories 改为 reactive 响应式数组

## 修复取消排除后值无法恢复的问题

### 问题：
勾选"不生成的名称"下拉框时，被排除行的 guige/shuliang 正确清零为 '0'。但取消勾选后，行重新显示，值却没有恢复为清零前的状态。

### 根本原因：
1. 旧逻辑在初始化时备份值（都是空字符串），排除时虽然会重新备份，但由于防抖 300ms 延迟，备份时机不准确
2. 恢复时用 `if (row.guige === '0')` 判断，但 computed 可能已经重新计算覆盖了值，导致条件不满足
3. 同名行（如两个"上包边"）共享同一个 mingcheng，遍历所有 tableData 时会影响所有同名行

### 解决方案：
重写排除/恢复逻辑，使用 `oldValue` 对比精确判断哪些是新增排除、哪些是取消排除：
- **新增排除时**：立即备份当前值到 Map，然后赋 '0'
- **取消排除时**：从 Map 恢复备份的值
- 去掉防抖，保证备份和恢复的时序准确
- 底部门板排除逻辑同理
